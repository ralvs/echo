# Graph analysis is one pure shared module over a neutral graph

`_shared/graph-analysis.ts` is the single home for algorithms over Echo's graphs — shortest path today, communities and centrality next. It operates on a deliberately minimal type, `WeightedGraph = { nodes: string[]; edges: { source; target; weight }[] }`: ids and weights, nothing else. It imports nothing (no graph library, no `db`, no `ai`), so it bundles trivially under Deno (ADR-0010) and every function is a pure unit test. Callers project their domain objects down to the neutral graph and map the returned ids back: `entity-graph.ts` (a sibling read projection of `entities` + `entity_edges`) exposes `entityGraph(db)` and `toWeightedGraph`, and the `find_path` MCP tool is the first formatting adapter over the pair.

## Why this is recorded

- **The graph model is id-only on purpose.** Forcing the algorithms to consume `RelationGraph` (relation-graph.ts) would drag thought-specific fields — `content`, `metadata`, `is_latest` — into a module that has no business knowing them, and would block reuse on the entity co-occurrence graph (undirected, integer-weighted, no relation semantics). The two graphs share zero columns. Keeping the algorithm input to ids + weights is the same "keep the graph model neutral, let the renderer slice it" discipline ADR-0015 records, taken one level deeper: here even the *node payload* is the caller's concern, not the module's.
- **This is the ADR-0016 move, not the ADR-0001 one — at first.** `find_path` is a single adapter, so on the "two adapters make a real seam" rule alone the algorithms could have lived in the tool. They don't, because — exactly like `lint.ts` — they are substantial branching logic (Dijkstra with deterministic tie-breaks, soon Brandes and label propagation) that is far better exercised against hand-built graph literals than through an MCP round-trip. Testability justifies the module on its own. The dashboard entity-graph view and `graph_overview` then arrive as genuine second and third adapters, but they are the bonus, not the reason.
- **Determinism is a contract, not an implementation detail.** Every traversal iterates in sorted-id order and breaks ties on the lower id. Without it, label propagation and equal-cost paths would reshuffle between identical calls — flaky tests and a digest that looks different every time you ask. The tie-break rule (`byId`) is one function the whole module shares.

## Consequences

- A new algorithm is one pure function plus a known-answer test; it needs no fake `db`/`ai` and no adapter to be verified.
- `find_path` loads the whole entity graph per call. That is fine for a personal corpus (hundreds of nodes) but is the first thing that will hurt at scale — before any algorithm's complexity does. The escape hatch is a bounded `entityEgoGraph(db, centerId, depth)` mirroring `egoGraph`; it is intentionally not built until the corpus demands it. `entityGraph` already accepts a `minWeight` floor to shed weak ties.
- `shortestPath` uses an O(V²) linear min-scan rather than a heap: obviously correct, deterministic, and instant at this scale. Swapping in a hand-rolled heap is a local change behind the same signature if profiling ever asks for it — never a graph library (that would break the Deno-bundle invariant).
